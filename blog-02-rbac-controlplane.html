The Mental Model Problem

Most engineers encounter Kubernetes RBAC by writing their first Role and RoleBinding to make a CI pipeline work — and then building a mental model around that experience. That mental model is incomplete in some critical ways.

The most damaging incomplete model is this: RBAC is primarily about human users accessing the cluster via kubectl. In reality, the majority of API requests in a production cluster come from software — controllers, operators, admission webhooks, sidecar agents, and CI systems. The RBAC design that governs those software identities matters more, and is more often neglected.

The Four Subjects — and Which One Gets Misconfigured Most

Regular users authenticate via certificates, OIDC tokens, or webhook authenticators. Kubernetes has no User API object — users exist only as subjects in certificate CN fields or token claims.

Service accounts are Kubernetes-native identities for pods and controllers. They're namespaced, they have automatically-generated API tokens, and they're what most workloads use to authenticate to the Kubernetes API. This is where most misconfiguration lives.

Groups are implicit collections, not explicit objects — defined by the Organization field in a user certificate, or system-level membership like system:nodes.

System identities — system:kube-scheduler, system:kube-controller-manager, system:nodes — are Kubernetes-internal subjects with pre-bound cluster roles.

The Real Failure Mode

The default ServiceAccount accumulation problem

Team A deploys a monitoring agent. It needs to list pods across namespaces, so they bind cluster-admin to the default ServiceAccount in the monitoring namespace. Six months later, another team deploys an application into the same namespace without specifying a serviceAccountName. That application's pod silently receives a token that grants cluster-admin. Nobody notices until an application vulnerability is exploited and the attacker uses the token to list every secret in the cluster.

Designing RBAC at Scale: The Patterns That Hold Up

Pattern 1: Namespace-per-team with standardized role tiers

Instead of crafting bespoke Roles for every workload, define three or four tiers at the cluster level — read-only, operator, deployer, admin — and bind teams to the appropriate tier in their namespace.

cluster-roles-tier-definitions.yaml YAML 
# Tier 1: Read-only — for dashboards, auditors, on-call engineers
kind: ClusterRole
metadata:
  name: tier-readonly
rules:
- apiGroups: ["", "apps", "batch"]
  resources: ["pods", "deployments", "jobs", "services", "configmaps", "events"]
  verbs: ["get", "list", "watch"]
# Explicitly NOT: secrets, exec, portforward, delete

---
# Tier 3: Deployer — for CI pipelines writing new workloads
kind: ClusterRole
metadata:
  name: tier-deployer
rules:
- apiGroups: ["apps"]
  resources: ["deployments", "statefulsets", "daemonsets"]
  verbs: ["get", "list", "create", "update", "patch"]
# No delete — prevent accidental or malicious destruction
Watch This

A RoleBinding that references a ClusterRole grants the ClusterRole's permissions only within the namespace of the RoleBinding. A ClusterRoleBinding referencing the same ClusterRole grants those permissions cluster-wide. The distinction is easy to miss; the blast radius difference is enormous.

The Control Plane: What's Actually Running and Why It Matters

Most Kubernetes security conversations treat the control plane as a monolith. In reality it's a collection of processes — each with its own API surface, authentication model, and failure modes. Understanding each component is the foundation of control plane hardening.

The API server: your biggest attack surface

The kube-apiserver is the single point through which all cluster state is managed. If you compromise it, you have the cluster. Beyond RBAC, the API server has configuration parameters frequently left at insecure defaults:

audit: kube-apiserver flags to verify SHELL 
# Inspect running API server arguments on a control plane node:
ps aux | grep kube-apiserver | tr ' ' '\n' | grep -E 'insecure|anonymous|audit|admission'

# Do NOT want to see:
--insecure-port=8080
--anonymous-auth=true

# Want to see:
--anonymous-auth=false
--audit-log-path=/var/log/k8s/audit.log
--audit-policy-file=/etc/kubernetes/audit-policy.yaml
--enable-admission-plugins=...NodeRestriction,...

etcd: the database you never think about until something explodes

etcd holds every secret, every configuration, every workload spec — in a format that requires only base64 decoding to read. Direct access to the etcd API bypasses Kubernetes RBAC entirely.

Real Threat Vector

The unauthenticated etcd endpoint

In several high-profile Kubernetes cluster compromises, the attack chain started with an exposed etcd port (2379/2380) reachable from the internet — or from a workload namespace — with no authentication. An attacker who can reach etcd can etcdctl get --prefix /registry/secrets and retrieve every secret in the cluster in seconds. No RBAC. No audit log. No admission webhook.

etcd-security-flags.conf CONF 
# Peer-to-peer TLS (between etcd cluster members)
--peer-cert-file=/etc/etcd/pki/peer.crt
--peer-key-file=/etc/etcd/pki/peer.key
--peer-client-cert-auth=true

# Client TLS (between API server and etcd)
--cert-file=/etc/etcd/pki/server.crt
--client-cert-auth=true

# Encryption at rest:
--encryption-provider-config=/etc/kubernetes/encryption.yaml
Encryption at Rest

Enabling mTLS for etcd communication does not encrypt data at rest. A cluster with etcd TLS configured but no EncryptionConfiguration still stores secrets as base64-encoded plaintext on disk. Enable aescbc or aesgcm provider for Secrets at minimum.

Building an Audit Log That's Actually Useful

A well-engineered audit policy is tiered. High-signal events — secret access, exec/attach, authentication failures, policy changes — get full RequestResponse logging. Routine read operations get Metadata. Internal system traffic gets None.

/etc/kubernetes/audit-policy.yaml YAML 
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
  # CRITICAL: Log all secret access in full
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["secrets"]

  # HIGH: Pod exec, attach, portforward
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["pods/exec", "pods/attach", "pods/portforward"]

  # LOW: Internal system traffic — suppress it
  - level: None
    users: ["system:kube-proxy", "system:node"]

  # DEFAULT: Metadata only for everything else
  - level: Metadata
Mature Posture

What "control plane hardening is done" actually looks like

Automated CIS benchmark runs in CI for cluster provisioning changes. Audit logs shipping to SIEM with real-time alerting on high-signal events. etcd network-isolated and mTLS-authenticated with encrypted-at-rest secrets. Kubelet anonymous auth disabled cluster-wide. Certificate rotation automated. NodeRestriction admission enabled. And crucially — documentation of every intentional deviation from default security settings, with an owner and a review date.

The final post in this series moves down from the cluster infrastructure layer into the workload layer — pod-level security, network isolation, and the runtime detection systems that catch what everything else misses.
