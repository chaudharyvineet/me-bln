From Supply Chain to Runtime:
The Full Arc of Container Image Security

April 2025 ~18 min read

Container image security isn't a single gate — it's a chain of decisions that begins at your Dockerfile and doesn't end until the last replica terminates. Senior engineers know that every gap in that chain is a trust assumption waiting to be violated.

Let's start with an honest framing: most Kubernetes clusters trust their container images far too much. The image was built somewhere, pushed to a registry, and that's the end of the scrutiny. Pull it, run it, done.

This posture made sense when organizations ran five Docker Compose stacks on VMs they personally provisioned. It makes no sense in a world where a platform team might be running hundreds of microservices built by dozens of different teams, pulling from half a dozen upstream registries, with base images that were last examined sometime last quarter.

The question isn't whether your image-layer risk is manageable. It's whether you actually know what's in your images — and whether anything is enforcing a policy about what's allowed to run.

The Trust Model Nobody Draws

Every container that runs in your cluster implicitly embodies a trust assertion: this image is safe to execute with these permissions, on these nodes, next to these other workloads. That assertion is almost never made explicitly. It accumulates through the absence of policy.

Consider the inheritance structure of a typical production image:

Image Layer Inheritance — Risk Propagation
🐧

ubuntu:22.04 — unpatched kernel libs, 90 days stale
→

node:18 — inherits all base CVEs plus npm tree

The uncomfortable truth here is that the layer your team wrote is usually the most secure part. The real exposure lives in the layers you inherited — the base OS packages, the language runtime, the transitive dependency tree that nobody has read. A vulnerability in any parent layer is present in every child image built on top of it.

⚠

The stale base image problem is worse than most teams realize. If your CI pipeline pins to node:18-slim and rebuilds weekly, you'll pick up security patches. If it pins to a digest and nobody updates it, your base layer ages in place. After six months, you're running an OS with dozens of known CVEs that upstream maintainers patched months ago.

What Image Scanning Actually Does — and Doesn't Do

Image scanning gets described in a way that implies more coverage than it actually provides. A scanner typically opens the image manifest, walks each layer, and inventories the installed software packages against known vulnerability databases (NVD, distro-specific advisories, language-ecosystem advisories).

What scanners examine

  • → Is the container configured to run as a specific non-root UID?
  • → Are there unnecessarily exposed ports in the image definition?
  • → Are SUID/SGID binaries present?
  • → Does the image use a pinned base digest or a floating tag?
  • → Are there secrets or credentials embedded in environment variables or layers?

What scanners cannot see

  • ✕ Runtime behavior — what the application actually does with its access
  • ✕ Logic vulnerabilities in application code
  • ✕ Zero-days — CVEs that haven't been published yet
  • ✕ Supply chain compromises injected upstream before the image was assembled
◆

Engineering a Hardened CI/CD Gate

A properly engineered scan gate has four properties: it runs at the right point in the pipeline, it has an authoritative policy that it actually enforces, it fails fast with actionable output, and it's exempt-free by default.

Secure CI Pipeline — Gate Position
📝
Code Push
Commit triggers CI pipeline
→
🔨
Build
docker build — image stays local
→
🔍
Scan Gate
CVEs + policy eval. Hard stop on failure.
→
📤
Push
Only policy-passing images reach registry
→
🚀
Deploy

.github/workflows/build.yml YAML 
- name: Build image
  run: |
    docker build \
      --build-arg BUILD_DATE=$(date -u +"%Y-%m-%dT%H:%M:%SZ") \
      --build-arg VCS_REF=${{ github.sha }} \
      -t $IMAGE_NAME:${{ github.sha }} .

- name: Scan image (hard gate)
  run: |
    scanner scan \
      --image $IMAGE_NAME:${{ github.sha }} \
      --fail-on CRITICAL,HIGH \
      --policy-file ./security/scan-policy.yaml \
      --output sarif --output-file scan-results.sarif

    # Pipeline hard-stops here if exit code != 0.
    # No push. No deploy. No exception without a ticket.

- name: Push to registry (only on scan pass)
  run: |
    docker push registry.internal/prod/$IMAGE_NAME:${{ github.sha }}
security/scan-policy.yaml YAML 
version: "2"
policies:
  # Hard failures — pipeline stops
  - action: FAIL
    rules:
      - cve_severity: [CRITICAL, HIGH]
        fix_available: true
      - base_image_tag: ["latest", "stable", "edge"]
      - run_as_root: true
      - suid_binary_found: true

  # Allowlist — suppressed findings with mandatory expiry
  allowlist:
    - cve_id: CVE-2024-XXXXX
      reason: "No upstream fix, mitigated by network policy"
      expires: "2025-06-01"
      approved_by: "security-team"
◈

The allowlist is not a get-out-of-jail-free card. Every suppressed finding needs an expiry date, a named approver, and a documented compensating control. An allowlist entry with no expiry is a debt that accumulates silently.

Inline Scanning vs. Registry Scanning: The Real Tradeoffs

Dimension Inline Scanning Registry Scanning
When it runs At build time, before push Continuously, against registry contents
What it catches New CVEs introduced in this build CVEs published after the image was built
Can prevent deployment Yes — pipeline gate Not directly — requires webhook integration
Value for aging images None — only run at build time High — catches CVEs disclosed post-build

Enforcing Trusted Image Provenance in the Cluster

Even with a solid scan gate, nothing prevents a developer from running kubectl run --image=python:3.11 and pulling directly from Docker Hub. All your CI work is bypassed in one command. The control that closes this gap is a ValidatingAdmissionWebhook.

registry-admission-webhook.yaml YAML 
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionWebhook
metadata:
  name: image-policy.security.internal
webhooks:
  - name: image-policy.security.internal
    rules:
      - apiGroups:   [""]
      operations:  ["CREATE"]
      resources:   ["pods"]
    failurePolicy: Fail  # Reject if webhook is unreachable
ℹ

Set failurePolicy: Fail, not Ignore. If your webhook service is unreachable, Ignore mode silently allows all images through. Fail mode blocks all pod creation until the webhook recovers. In a security-sensitive cluster, that's the right default.

For production deployments, your CD pipeline should resolve tags to digests before writing the pod spec. Tags are mutable — registry.internal/myapp:1.4.2 can be overwritten at any time. Digest references are immutable and give you cryptographic proof that what's running is what was scanned.

✦

The full loop matters. Your security posture is only as strong as the weakest link: build → scan → push → admit → runtime. A hardened CI gate with no admission webhook is defeated by one direct kubectl apply. Don't optimize one layer to the exclusion of others.
