There's a thought experiment worth running when you onboard a new workload: if an attacker were to achieve arbitrary code execution inside this container — through an application-layer vulnerability — what's the blast radius? What can they access? What can they do to other services? Where can they go from here?

The controls in this post are how you make that blast radius small.

Security Contexts: Constraining What the Container Can Do

A Security Context is the per-pod, per-container declaration of the security posture Kubernetes will enforce. If you're running containers without Security Contexts, you're running with defaults — and the defaults are not conservative.

pod-security-context-production.yaml YAML 
apiVersion: v1
kind: Pod
spec:
  # Pod-level: applies to all containers
  securityContext:
    runAsUser:          10001
    runAsGroup:         10001
    runAsNonRoot:       true
    seccompProfile:
      type: RuntimeDefault

  containers:
  - name: api
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem:   true
      capabilities:
        drop:
        - ALL  # Drop everything first, add back only if needed

The readOnlyRootFilesystem: true setting deserves emphasis. If an attacker achieves code execution in the container, they cannot modify binaries, write a persistence mechanism, or drop a payload to disk. Everything they do is ephemeral — when the container restarts, their changes are gone.

⚠
The capability drop gotcha

Dropping ALL capabilities and then adding back what's needed is the correct order. Many engineers leave the default set intact and add additional capabilities. The default set includes NET_RAW, CHOWN, AUDIT_WRITE, and others that most applications have no business using. Always drop ALL, then add back the minimum required set explicitly.

Admission enforcement: Pod Security Admission

Defining Security Contexts in individual pod specs is insufficient — developers will forget them. The enforcement layer is a cluster-level policy that rejects non-compliant pod specs at admission time. PSA applies automatically based on namespace labels:

namespace-psa-enforcement.yaml YAML 
apiVersion: v1
kind: Namespace
metadata:
  name: payments
  labels:
    # Enforce: reject non-compliant pods at admission
    pod-security.kubernetes.io/enforce:         restricted
    pod-security.kubernetes.io/enforce-version:  latest
    # Audit without rejecting (useful for legacy migration)
    pod-security.kubernetes.io/audit:           restricted

Network Policies: The Firewall You Own

In a default cluster — no network policies applied — every pod can reach every other pod in the cluster, across namespaces, regardless of what they are. A compromised frontend can reach your database. Network policies change this by imposing explicit, label-based allow rules. Traffic not explicitly allowed is implicitly denied — once you apply a policy.

network-policy-payments-namespace.yaml YAML 
# Step 1: Default deny all in namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: payments
spec:
  podSelector: {}   # Matches all pods
  policyTypes: [Ingress, Egress]
# No rules = deny all

---
# Step 2: Allow API to reach Postgres only
spec:
  podSelector:
    matchLabels:
      app: postgres
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api-server
    ports:
    - protocol: TCP
      port: 5432
◈
CNI Support is Required

NetworkPolicy resources are only enforced if your CNI plugin supports them. Flannel does not. Calico, Cilium, and Weave Net do. Create a NetworkPolicy, then verify enforcement by attempting a blocked connection from inside the cluster. If it succeeds, your CNI is silently ignoring the policy.

Behavioral detection rules — context is everything

Effective runtime detection rules need to be contextually aware — not just "was this syscall called?" but "was it called by this process, in this container image, in this Kubernetes namespace, at this time?" Rules that lack context generate noise. Rules that have context generate signal.

behavioral-detection-rules.yaml Falco Syntax 
# Rule: shell spawned inside container (high signal for api services)
- rule: Shell execution in container
  condition: |
    evt.type = execve
    and container.id != host
    and proc.name in (bash, sh, zsh, dash)
    and not container.image.repository in (known_debug_images)
  output: |
    Shell spawned: user=%user.name container=%container.name
    image=%container.image.repository cmd=%proc.cmdline
    ns=%k8s.ns.name pod=%k8s.pod.name
  priority: WARNING

---
# Rule: unexpected outbound from database pod
- rule: Unexpected outbound from database
  condition: |
    evt.type = connect
    and container.label[tier] = database
    and not fd.sip in (allowed_cluster_ips)
  priority: ERROR

Automated response: closing the loop

A detection that fires an alert and waits for a human to respond has a response time measured in minutes to hours. Closing the loop means wiring detection events to automated response playbooks.

Trigger Event
Automated Response
Shell spawned in production pod
Capture forensic state → Cordon pod → Page on-call with container snapshot
kubectl exec outside business hours
Alert security channel → Revoke SA token if not acknowledged in 5 min
Outbound to known C2 IP
Patch NetworkPolicy to isolate pod → Page on-call as P0
Privilege escalation syscall
Kill pod → Capture image digest → Block image in registry
◑
On False Positives

Automated response is only safe when your detection rules are well-tuned. A false positive that kills a production pod is its own incident. Run new rules in observe-only mode for at least two weeks before enabling automated response. Track false positive rate explicitly.

Each layer of this series addresses a different part of the threat model — they're not redundant, they're complementary. An attacker who bypasses image scanning still faces RBAC. An attacker who escalates to cluster-admin still has a container with a read-only filesystem. An attacker who escapes the container trips a behavioral rule before they can move laterally. No single layer is the answer. The answer is the stack.
